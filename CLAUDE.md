# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Splity is a one-shot JVM CLI that keeps a shared "split" account in sync between two people's separate YNAB ([You Need A Budget](https://ynab.com)) budgets. Each transaction in one budget gets a mirrored **complement** (same date, negated amount) in the other, and Splity reconciles creates/updates/deletes across both on every run. It's meant to be run on a schedule (e.g. cron); `main()` does a single sync pass and exits.

## Commands

All commands run from the repo root via the Gradle wrapper. JDK 17 (see `.sdkmanrc`).

- **Full check** (compile + lint + test, same as CI): `./gradlew build`
- **Run the sync** against `./config.yaml` in the working directory: `./gradlew run`
- **Package a distributable**: `./gradlew assemble` → unzip `build/distributions/splity-<version>.zip`, run `bin/splity`
- **Test**: `./gradlew test`
- **Single test class**: `./gradlew test --tests "com.ryanmoelter.splity.TransactionMirrorerTest"`
- **Lint**: `./gradlew lintKotlin` — **Format**: `./gradlew formatKotlin` (kotlinter/ktlint; `build` fails on unformatted code). Generated code under `generated/` is excluded.
- **Regenerate the YNAB API client**: `./gradlew :ynab-api:generateSwagger`, then apply the manual fixups in [ynab-api/Generating the API.md](ynab-api/Generating the API.md). The generator is imperfect — don't hand-edit generated files expecting them to survive a regen.

Config lives in a `config.yaml` in the working directory (**not** committed-as-canonical — see [README.md](README.md) for the schema: YNAB token, the two budget/account name pairs, optional `startDate` and Sentry DSN). `Config.version` is *not* read from yaml — it's injected at build time from `version.properties` (generated by the `createProperties` Gradle task) via Hoplite's `PropertySource`.

## Modules

- **`:` (root)** — the sync engine, package `com.ryanmoelter.splity`.
- **`:ynab-api`** — a mostly-generated Retrofit + Moshi client for the YNAB v1 REST API (package `com.ynab.client`). Generated from `ynab-api/ynab.json` via the Yelp codegen plugin. The `YnabClient` interface exposes `budgets`/`accounts`/`transactions`/`categories` APIs.

## The sync pipeline

`TransactionMirrorer.mirrorTransactions()` ([TransactionMirrorer.kt](src/main/kotlin/com/ryanmoelter/splity/TransactionMirrorer.kt)) orchestrates four stages, each wrapped in a Sentry span:

1. **`Repository.fetchNewTransactions()`** — incremental delta fetch from both budgets using `serverKnowledge` cursors persisted in the `SyncData` row, written into a local SQLite cache. On first run it resolves budget/account IDs from the config *names* and persists `SyncData`. Each fetched transaction is diffed against what's already stored and tagged with a `ProcessedState`.
2. **`TransactionSplitter.splitFlaggedTransactions()`** — finds PURPLE-flagged transactions in *non-split* accounts and splits them 50/50 into the split account (payer keeps the odd cent), then re-fetches.
3. **`ActionCreator.createDifferentialActionsForBothAccounts()`** — the core logic. Matches each unprocessed transaction with its complement in the other budget, then emits a `TransactionAction` based on the **cross-product of both sides' `ProcessedState`** (see the nested `when` in `createAction`).
4. **`ActionApplier.applyActions()`** — executes each action against the YNAB API and updates the local cache.

### Key concepts

- **`ProcessedState`** (`UP_TO_DATE` / `CREATED` / `UPDATED` / `DELETED`) — the lifecycle marker on every stored transaction. After a clean sync everything is `UP_TO_DATE`; the fetch stage re-tags changed rows, action creation reads those tags, and `Repository.markProcessed` resets them. This is the spine of the whole design — most logic branches on it.
- **Complement matching** — a complement is found by date + negated amount. Because a transaction may have *changed* since it was last matched, the matcher first looks in the **`ReplacedTransaction`** snapshot (the pre-change version) so it can still pair things up by their last-synced values.
- **`ReplacedTransaction` / `ReplacedSubTransaction`** — when an `UP_TO_DATE` transaction changes, its prior version is snapshotted into these tables. Used both to re-find complements and to compute exactly which fields changed (`UpdateField`, see [ReplacedTransactionExtensions.kt](src/main/kotlin/com/ryanmoelter/splity/database/ReplacedTransactionExtensions.kt)).
- **`TransactionAction`** (sealed interface in [ActionApplier.kt](src/main/kotlin/com/ryanmoelter/splity/ActionApplier.kt)) — `CreateComplement` / `UpdateComplement` / `DeleteComplement` / `MarkProcessed` / `MarkError`. `MarkError` flags a transaction RED in YNAB with a human-readable memo when sync can't safely proceed (e.g. both sides edited).

## Persistence (SQLDelight)

Local cache is SQLDelight 1.5.5; `.sq` files in `src/main/sqldelight/com/ryanmoelter/ynab/` generate the `Database` (package `com.ryanmoelter.ynab.database`, `verifyMigrations = true`). Production uses a file DB `incrementalSyncCache.db`; tests use an in-memory driver. All column adapters (enum + the value-class IDs) are wired up in [DatabaseModule.kt](src/main/kotlin/com/ryanmoelter/splity/database/DatabaseModule.kt).

IDs are `@JvmInline value class`es ([Ids.kt](src/main/kotlin/com/ryanmoelter/splity/database/Ids.kt)): `BudgetId`/`AccountId`/`CategoryId`/`PayeeId` wrap `UUID`; `TransactionId`/`SubTransactionId` wrap `String`. The internal domain model is `PublicTransactionDetail` / `PublicSubTransaction` — conversions to/from both the API's `TransactionDetail` and the stored `StoredTransaction` live in the `database/*Extensions.kt` files.

## Dependency injection (kotlin-inject)

DI is compile-time via kotlin-inject + KSP — components are built through generated `.create()` extension functions (e.g. `SplityComponent::class.create(...)`). `SplityComponent` aggregates four modules, each with a `Real*` (production) and `Fake*` (test) implementation:

| Module | Real | Fake |
|---|---|---|
| `DatabaseModule` | file SQLite | in-memory SQLite |
| `ConfigModule` | loads `config.yaml` | hardcoded test config |
| `ApiModule` | real `YnabClient` over OkHttp | `FakeYnabClient` |
| `SentryModule` | Sentry | no-op |

## Testing

Tests are **integration tests of the whole pipeline**, not unit tests. They drive `TransactionMirrorer` against a `FakeYnabServerDatabase` (an in-memory model of YNAB's server) plus an in-memory SQLDelight DB, then assert end state with custom matchers in `src/test/kotlin/.../test/AssertionExtensions.kt`. Stack: Kotest `FunSpec` (JUnit5 platform) + Strikt + MockK. `KotestProjectConfig` sets `IsolationMode.InstancePerLeaf` and random test order, so each leaf test gets a fresh fixture — don't rely on cross-test state.

## Conventions

- 2-space indent, 100-col max ([.editorconfig](.editorconfig)). Single-name imports (no star imports), imports sorted with aliases last.
- ktlint's `function-naming` rule is disabled (composable-style names allowed).
- Compiled with `-progressive` and the JVM target is 11 — `when` over a sealed type / enum is treated as needing to be exhaustive; the codebase leans on this heavily (exhaustive `when (processedState)` blocks). Don't add `else` branches that would mask a missing case.
